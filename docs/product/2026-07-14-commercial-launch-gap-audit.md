# 商业上线缺口审计

## 审计结论

当前产品已经具备可验证的免费价值和初步 Pro 价值，但还不是可公开收款的商业服务。

已经成立的基础包括：本地优先记录、训练执行、阶段复盘、数据导入导出、账号身份边界、服务器验证权益、原子 AI 配额，以及 90 天和年度 Pro 报告。当前最主要的缺口不再是多做一张图表，而是建立真实的生产服务、用户支持、账号生命周期和交易闭环。

在这些条件完成前，产品应继续保持没有购买按钮、没有浏览器可信 Pro 状态、没有假支付或手工公开开通入口。

## 当前证据

| 范围 | 当前状态 | 证据 | 结论 |
| --- | --- | --- | --- |
| 公开交付 | GitHub Pages 正常提供静态应用 `v1.17.0` | 公开 `app.js` 返回版本 `1.17.0` | 可交付免费本机模式 |
| 正式登录 | 公开地址的账号 API 返回 `404` | GitHub Pages 不运行 Node 服务 | 公开地址不能登录是部署边界，不是用户密码问题 |
| 权益 | Node 部署支持服务端验证 Free/Pro 和月度 AI 配额 | `server/entitlements.js`、Supabase migration、配额测试 | 权益技术基础已具备 |
| Pro 价值 | Pro 可在本机生成 90 天和年度报告 | `public/app.js`、浏览器冒烟测试 | 已有第一项可信付费价值 |
| 支付 | 没有订单、签名 webhook、退款或续费处理 | README 和条款明确当前无支付 | 不可公开收费 |
| 账号删除 | 只有登录和退出，没有认证账号删除流程 | `server.js` 账号路由 | 付费上线前必须补齐 |

| 生产监控 | 只有结构化请求日志和健康检查 | `server.js` | 缺少外部可用性监控与告警接收人 |

## 上线门槛

### P0：任何公开收款之前

1. **确认经营与交易事实**
   - 明确经营主体、注册地区、目标市场、币种、月度和年度价格。
   - 明确试用、自动续费、取消、退款、发票和争议处理规则。
   - 选择支持目标市场和 Apple Pay 的支付服务商或收单机构，并完成商户与域名验证。
   - 验收证据：商户账户可用、正式条款可访问、结算账户验证完成、测试与生产凭据分离。

2. **部署正式同源后端**
   - 使用独立 HTTPS 域名运行 Node 服务，并配置 Supabase Auth、service role 和 OpenAI 服务。
   - GitHub Pages 可继续作为免费演示，但不能作为账号和支付主站。
   - 验收证据：正式域名的 `/api/health`、登录、会话刷新、权益读取和退出均通过端到端测试。

3. **补齐账号生命周期**
   - 提供认证账号删除流程，说明本机记录与云端账号分别删除。
   - 删除前处理有效订阅、待退款、配额事件和法定保留数据；不得把“退出登录”描述为“删除账号”。
   - 验收证据：登录用户可发起删除，重新登录失败或得到明确状态，相关政策与客服流程一致。

4. **建立真实支付和权益闭环**
   - 服务器创建结账会话；客户端返回页只显示待确认。
   - 仅由签名验证、幂等处理的 webhook 更新权益。
   - 覆盖付款成功、付款失败、取消续费、到期、退款、争议和恢复订阅。
   - 验收证据：每个状态均有服务端测试、重复事件测试和可核对的权益结果。

5. **提供真实支持渠道**
   - 在产品、隐私政策、条款和付款页面显示同一个支持入口。
   - 明确账号删除、退款、技术故障和隐私请求的处理方式及响应时间。
   - 验收证据：未登录用户也能访问；测试邮件或工单可被负责人收到并回复。

### P1：首批付费用户之前

1. **发布门禁**
   - 在部署前运行语法检查、服务端权益测试和浏览器冒烟测试。
   - 失败时不得发布；生产版本号必须与构建版本一致。

2. **监控与故障响应**
   - 对首页、健康检查、登录、权益和支付 webhook 建立外部监控。
   - 告警必须有接收人、升级方式和最低日志保留周期。
   - 日志不得包含认证令牌、原始健康记录、备注或支付密钥。

3. **邮件送达能力**
   - 配置正式发信域名、SPF、DKIM 和 DMARC，验证验证码模板和到达率。
   - 对发送失败、频率限制和验证码过期提供清晰恢复路径。

4. **政策与第三方披露**
   - 写明经营主体、联系方式、支付方、认证方、AI 服务方、数据范围、保存期限和用户权利。
   - 深度解读上线时单独说明只发送聚合数据，并提供独立撤回授权。

5. **付费状态体验**
   - 明确展示 Free、Pro、取消待到期、付款失败、退款、权益暂不可用等真实状态。
   - 任何异常都不得锁住本机记录、免费报告、安全提示和完整导出。

## 产品优化顺序

### 近期：Pro 长期报告云端深度解读

这是当前最适合继续开发的付费价值：它建立在已完成的长期报告和 AI 配额上，能够产生真实云成本，也不需要先托管完整训练历史。

推荐边界：

- 只允许服务器验证的 Pro 用户调用。
- 只发送 90 天或年度报告的聚合指标、确定性发现和行动建议。
- 不发送日期、日记、训练明细、动作、重量、次数、RPE、备注或本地 ID。
- 使用独立授权，不继承近期云端建议授权。
- 使用现有月度 AI 配额；失败释放预留，额度耗尽仍保留完整本地报告。
- 第一版结果只保留在当前页面，不建立新的云端或本地历史。

### 中期：可信恢复而非立即全量同步

多设备同步有明显付费价值，但也会显著扩大健康数据托管责任。应先验证用户是否愿意为长期复盘付费，再设计端到端的数据模型、冲突处理、加密、删除和恢复策略。不能把浏览器 `localStorage` 直接镜像到数据库当作同步方案。

### 后期：基于证据调整套餐

在正式付费漏斗可测量前，不应凭想象继续增加 Pro 功能。首批指标应聚焦：

- 完成首条记录、首场训练和首份 28 天报告的比例。
- 达到长期报告门槛的时间与比例。
- 查看 Pro 价值后进入结账、完成支付和次月续费的比例。
- 云端深度解读的使用率、失败率、单位成本和主动撤回率。
- 退款、账号删除、支持请求和高风险建议反馈。

分析应优先使用匿名聚合事件，不采集训练内容或健康记录。事件采集本身也必须在隐私政策中如实披露。

## 暂缓事项

- “支持伙伴”继续停止开发。
- 不在缺少商户资料时制作支付按钮或订阅假状态。
- 不以云同步换取短期功能数量；先补齐删除、支持和事故响应能力。
- 不把安全提醒、基础记录、JSON 导出或既有本机记录变成付费墙。

## 下一次需要经营方确认的信息

- 经营主体和注册地区。
- 首发国家或地区、币种和税务处理方式。
- 月度与年度定价、试用、退款和自动续费规则。
- 支付服务商与 Apple Pay 可用性。
- 正式域名、支持邮箱或工单地址、结算账户。
- 隐私请求和账号删除的责任人及响应时限。

这些信息缺失不妨碍继续开发聚合深度解读和发布门禁，但会阻止真实支付、正式条款和公开商业上线。
