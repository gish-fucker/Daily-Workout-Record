# P0 健身新手首次训练体验实施计划

日期：2026-07-15  
依据：`2026-07-15-p0-beginner-training-experience-design.md`  
范围：只实施已批准的 P0；不夹带 P1、P2 功能

## 实施原则

1. 先建立显式训练会话模型和自动化证据，再改界面。
2. 每个阶段保持现有历史记录、导入导出和离线能力可用。
3. 模板目标、实际结果和完成状态不得再次合并或通过输入变化推断。
4. 每完成一个垂直切片即运行对应检查，避免最后一次性排查。
5. 只提交本计划涉及的文件，不纳入现有未跟踪文档、反馈文件或运行日志。

## 任务 1：建立显式训练会话模型与单元测试

涉及文件：

- 新增 `public/workout-session-model.js`
- 新增 `scripts/workout-session-test.mjs`
- 修改 `package.json`

实现：

1. 定义版本化训练草稿结构：`version`、`startedAt`、`currentSetId`、训练标题、动作、组目标、实际结果和 `pending/completed/skipped` 状态。
2. 定义计量类型：`reps`、`seconds`、`minutes`、`completion`；重量始终可选。
3. 实现纯函数：创建会话、完成组、撤销、跳过、选择下一组、统计进度、判断能否结束、转换为正式训练记录。
4. 实现旧草稿迁移：保留原输入；旧规则能确认的组标记为 `completed`，其余为 `pending`；迁移结果必须允许用户修改。
5. 将“轻松、正合适、有点吃力”映射到现有整体 RPE，但不提供默认值。
6. 浏览器以普通脚本加载模型并暴露单一命名空间，保持现有 `app.js` 和浏览器冒烟可继续访问原全局函数。
7. 在 `package.json` 中新增独立会话测试命令，将模型语法检查和会话测试纳入现有检查链路。

测试证据：

- 模板默认次数未修改，显式完成后可以输出正式组。
- 自重、秒、分钟、仅完成组不填写重量仍可完成。
- 只输入数据但未完成的组不输出。
- `skipped` 组不输出。
- 完成、撤销、跳过后的下一组选择正确。
- 没有完成组时不能生成正式训练。
- 旧草稿迁移保留输入并产生可见状态。

检查命令：

- `node --check public/workout-session-model.js`
- `node scripts/workout-session-test.mjs`

提交检查点：`Add explicit workout session model`

## 任务 2：拆分落地页和应用路径

涉及文件：

- 修改 `public/index.html` 为极简落地页
- 新增 `public/app/index.html`，承载现有应用壳
- 修改 `server.js`
- 修改 `public/manifest.webmanifest`
- 修改 `public/sw.js`
- 修改 `scripts/smoke-test.mjs`
- 修改 `README.md`

实现：

1. 将现有应用结构迁移到 `public/app/index.html`，共享资源使用兼容 GitHub Pages 仓库子路径的相对地址。
2. 调整应用内脚本、样式、图标、Manifest、隐私和条款链接，使它们从 `/app/` 和仓库子路径均能正确解析。
3. 根路径只保留产品承诺、三步价值、可信本机存储说明和“看看今天怎么练”主按钮。
4. 本地服务将 `/app` 与 `/app/` 映射到应用入口；静态文件和 API 行为保持不变。
5. Manifest 更新显示名称、副标题和 `start_url`，安装后直接打开 `/app/`。
6. Service Worker 缓存落地页、应用页和共享资源；根路径与应用路径分别使用正确的离线回退。
7. 更新缓存版本，确保旧 Service Worker 不继续提供旧单页入口。
8. README 明确公共落地页、应用入口、本机存储和 PWA 启动路径。

测试证据：

- `/` 返回落地页，`/app` 与 `/app/` 返回应用。
- 落地页主按钮在仓库子路径部署下正确进入应用。
- Manifest 在 `/Daily-Workout-Record/` 子路径下解析到 `/Daily-Workout-Record/app/`。
- 离线根路径回退落地页，离线应用路径回退应用页。
- API、隐私页和条款页不受影响。

检查命令：

- `node --check server.js`
- `npm run check`
- 扩展后的静态路由冒烟

提交检查点：`Split landing and app routes`

## 任务 3：收束产品名称、应用页头与首次首页

涉及文件：

- 修改 `public/app/index.html`
- 修改 `public/app.js`
- 修改 `public/styles.css`
- 修改 `public/privacy.html`
- 修改 `public/terms.html`
- 修改 `public/manifest.webmanifest`
- 修改 `scripts/smoke-test.mjs`

实现：

1. 用户可见名称统一为“今天练什么”，副标题统一为“根据恢复状态安排适合今天的训练，并带你逐组完成”。
2. 页头只保留产品名称和设置入口。
3. 零训练记录首页只渲染 Home Decision：默认方案、一个主按钮、状态调整次入口，以及必要的异常疼痛说明。
4. 隐藏零数据状态下的周目标、支持约定、状态评分、完整日常表单、空洞察卡片和重复新手引导。
5. 有数据时保持现有功能可访问，但不把 P1 的继续训练、进步反馈或提醒扩展进本任务。
6. “为什么这样建议”改为默认折叠。
7. 正常页面不再展示账号、建议模式、离线和安装状态标签。

测试证据：

- 零数据 `/app/` 首屏只有一个视觉最强按钮。
- 390px 手机视口无需滚动即可看到方案和主按钮。
- 页面文本不再出现 `Personal log`、`Daily Coach` 等首页英文眉题。
- 周目标、支持约定和完整日常表单在首次核心路径中不可见。
- 已有记录、洞察、模板和帮助仍可访问。

提交检查点：`Simplify the first-use home`

## 任务 4：实现四项快速状态与疼痛安全门槛

涉及文件：

- 修改 `public/app/index.html`
- 修改 `public/app.js`
- 修改 `public/styles.css`
- 修改 `scripts/smoke-test.mjs`

实现：

1. 新增单屏 Quick Readiness 面板：睡眠、精力、酸痛和异常疼痛均使用普通语言大按钮。
2. 将选项映射到现有状态数值并保存当天记录；饮水、心情、习惯和备注放入“补充今日状态”。
3. 未填写四项时明确显示“新手默认方案”，不得暗示个性化。
4. 填写后重新计算建议，并显示“已根据今天状态调整”。
5. 当天缺少疼痛状态时，开始训练只经过一次二选一 Pain Gate；已有疼痛状态时不重复询问。
6. 明显疼痛不得载入普通负重方案，改为恢复方案并显示必要安全说明。
7. 焦点进入、关闭、错误提示和键盘操作符合现有对话框模式。

测试证据：

- 无状态时最多两次点击进入第一组。
- 已有疼痛记录时一次点击进入对应方案。
- 四项状态在 390px × 844px 视口内完成，无横向溢出。
- 选择不同状态能改变正常练、轻量练或恢复日。
- 补充字段不会重新占据首页核心路径。

提交检查点：`Add optional readiness adjustment`

## 任务 5：接入聚焦式逐组训练界面

涉及文件：

- 修改 `public/app/index.html`
- 修改 `public/app.js`
- 修改 `public/styles.css`
- 修改 `scripts/smoke-test.mjs`

实现：

1. 首页主按钮将推荐模板转换为版本化 Workout Session，并直接显示第一组。
2. 训练首屏显示训练名、真实开始时间计时、完成进度、当前动作、当前组、普通语言提示和匹配计量类型的控件。
3. “完成这一组”更新显式状态、持久化草稿、进入下一组并提供撤销。
4. “暂时跳过”只更新草稿，不写入正式记录。
5. 可展开完整训练计划，允许跳转任意组、返回修改和重新设为未完成。
6. 单组 RPE、备注和计划编辑进入“更多”，不占据默认首屏。
7. 现有自由训练和模板编辑能力保留为二级入口，不与当前组主操作竞争。
8. 草稿在输入、完成、撤销、跳过和切换后立即保存；刷新恢复当前组和计时。

测试证据：

- 模板默认值不修改即可完成。
- 平板支撑显示秒，快走显示分钟，自重动作不要求重量。
- 输入但未完成的组保持 `pending`。
- 完成后自动进入下一组并能撤销。
- 刷新后恢复开始时间、当前组、实际结果和全部状态。
- 390px 手机首屏可见当前组和完成按钮。

提交检查点：`Add focused set-by-set training`

## 任务 6：重构结束、总结与正式保存

涉及文件：

- 修改 `public/app/index.html`
- 修改 `public/app.js`
- 修改 `public/styles.css`
- 修改 `scripts/smoke-test.mjs`

实现：

1. 训练中不显示日期、标题、总时长、整体 RPE 和总结备注。
2. 结束时检查完成组数量：零完成组只能继续、保留草稿或放弃；存在未完成组时先提示剩余数量。
3. 总结页显示自动时长、必选整体感受、可选精确 RPE 和可选备注。
4. 使用会话模型转换正式训练，仅保存 `completed` 组；未修改模板目标作为实际结果写入。
5. 保存失败保留当前会话和输入，并提供明确重试入口。
6. 保存成功后清除草稿、更新历史并只显示简洁成功确认，不加入 P1 完成页。
7. 编辑旧训练时将既有组载入为 `completed`，保持替换保存而非重复新增。

测试证据：

- 空训练无法保存。
- 未完成组不会进入正式记录。
- 默认模板组、徒手组、秒和分钟组均正确保存。
- 未选择整体感受时不能提交，但精确 RPE 和备注可不填。
- 保存失败不丢草稿，成功后才清除。
- 编辑、删除和“填入上次”回归通过。

提交检查点：`Save only explicitly completed sets`

## 任务 7：收纳技术状态并保留异常提示

涉及文件：

- 修改 `public/app/index.html`
- 修改 `public/app.js`
- 修改 `public/styles.css`
- 修改 `scripts/smoke-test.mjs`

实现：

1. 将账号、建议来源、离线、安装和导出移动到现有帮助或设置区域。
2. 设置入口直接定位到该区域，不新增一级导航。
3. 文案改为用户语言：“数据保存在这台设备”“可以离线记录”“安装到主屏幕”“导出备份”。
4. 仅在离线、保存失败、存储异常或新版本可用时显示全局提示。
5. 正常在线、缓存已就绪等状态保持安静。

测试证据：

- 正常首页不存在技术状态胶囊。
- 设置区域仍可查看账号、安装、离线和导出能力。
- 模拟离线时只出现简短说明，训练和本地保存仍可用。
- 恢复在线后异常提示正确消失。

提交检查点：`Move technical status into settings`

## 任务 8：兼容、无障碍与视觉收尾

涉及文件：

- 修改 `public/app.js`
- 修改 `public/styles.css`
- 修改 `scripts/smoke-test.mjs`
- 必要时修改 `README.md`

实现：

1. 验证旧历史记录、旧草稿、模板、Strong/Hevy 导入和 JSON/CSV 导出。
2. 新计量类型进入 JSON 导出；缺少类型的旧数据按 `reps` 读取并保留原备注。
3. 完成状态同时使用文字、图标和颜色，不依赖单一视觉信号。
4. 自动进入下一组时移动焦点并使用礼貌状态提示；折叠计划、状态选项和错误信息具备无障碍名称。
5. 所有触控目标适合手机操作，并遵循减少动态效果设置。
6. 检查桌面、390px 手机、较短手机视口和大字号场景，无横向溢出或主按钮被遮挡。

测试证据：

- 键盘可完成状态调整、疼痛确认、逐组完成和保存。
- 屏幕阅读顺序与视觉顺序一致。
- 旧数据与导入导出回归通过。
- 桌面和手机截图无重叠、截断、空白大卡片或多主按钮竞争。

提交检查点：`Polish P0 accessibility and compatibility`

## 任务 9：完整验证与完成审计

检查范围：

1. 运行 `npm run check`。
2. 运行训练会话单元测试。
3. 运行现有权益测试。
4. 运行扩展后的完整浏览器冒烟测试。
5. 使用真实浏览器分别验证全新用户、已记录状态用户、明显疼痛用户、旧数据用户和离线用户。
6. 捕获桌面与手机的落地页、应用首页、四项状态、当前组和训练总结截图并逐张检查。
7. 检查 `/`、`/app/`、Manifest、Service Worker、隐私页和条款页的 HTTP 响应。
8. 检查 `git diff --check` 和工作区，只提交 P0 文件。
9. 对照设计文档第 16 节逐项记录证据；任何缺失证据视为未完成。

最终通过标准：

- 设计文档中的所有 P0 验收项均有自动化或真实浏览器证据。
- 没有 P1、P2 功能混入变更。
- 现有历史、编辑、删除、模板、导入导出、洞察和离线能力没有回归。
- 本地完成后不自动推送或发布；如用户要求发布，再推送并验证实际公共地址的 `/` 与 `/app/`。

提交检查点：`Complete P0 beginner training experience`
