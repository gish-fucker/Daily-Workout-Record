# 账号身份与数据边界设计

## 目标

为未来订阅权益、AI 配额和可选云同步建立可信的用户身份，同时保持当前产品的本地优先承诺。登录只证明“这个人是谁”，不表示本机训练或健康记录已经上传、备份或同步。

本阶段不展示价格、订阅按钮、Pro 状态或云同步开关，也不恢复“支持伙伴”开发。支付仍等待经营主体、商户账户、价格和退款政策。

## 方案选择

采用“Supabase 托管邮箱验证码 + 应用服务端同源代理 + HttpOnly Cookie”。

未采用：

- 自建密码账户：需要可靠数据库、邮件验证、密码哈希、找回与风控，当前运行环境不足以安全承担。
- 浏览器本地昵称：不能跨设备证明身份，也不能作为服务端权益归属依据。
- 浏览器直接保存 Supabase 会话：实现更快，但刷新令牌会暴露给页面脚本，不符合后续付费权益的安全基础。

Supabase 只负责身份。应用未来可以用 Supabase 用户 ID 关联服务端权益，但本阶段不创建训练数据表。

## 用户状态

账号区域放在“帮助”页，并在页头显示简短状态。

### 未配置

- 状态显示“本机模式”。
- 解释当前部署没有连接账号服务。
- 不显示无效登录表单。
- 本地记录、导出、安全提示和离线能力正常可用。

### 未登录

- 显示邮箱输入和“发送验证码”。
- 明确验证码用于建立身份。
- 明确登录不会上传当前设备记录。

### 等待验证码

- 显示已发送提示和一次性验证码输入。
- 邮箱只在当前页面内存中保留，不写入应用本地记录或备份。
- 用户可以返回修改邮箱。

### 已登录

- 显示经过遮罩的邮箱和“身份已连接”。
- 显示“本机数据未同步”。
- 提供退出按钮。
- 不显示订阅或权益状态。

### 离线或服务错误

- 已有本地功能继续工作。
- 账号区域说明暂时无法连接，不把网络错误解释为账号不存在。
- 不清空本地记录。

## 服务端接口

### `GET /api/account/session`

未配置时返回 `{ configured: false, signedIn: false }`。配置后读取 HttpOnly Cookie，验证访问令牌；访问令牌失效且存在刷新令牌时进行一次刷新并轮换 Cookie。成功只返回用户 ID、遮罩邮箱所需的邮箱和会话状态，不返回令牌。

### `POST /api/account/request-code`

接受 `{ email }`，校验长度和邮箱结构后调用 Supabase `/auth/v1/otp`。响应不透露邮箱是否已注册。按来源地址和操作类型限流。

### `POST /api/account/verify`

接受 `{ email, token }`，验证码只允许 6 至 8 位数字。调用 Supabase `/auth/v1/verify`，成功后设置访问与刷新 Cookie，并只返回安全用户摘要。

### `POST /api/account/signout`

尽力通知 Supabase 注销当前访问令牌，无论上游结果如何都清除本应用 Cookie。退出不删除本机记录。

## Cookie 与请求安全

- Cookie 使用 `HttpOnly`、`SameSite=Strict`、`Path=/`。
- HTTPS 生产环境增加 `Secure`；本机 HTTP 测试不增加，保证开发可用。
- 访问 Cookie 按上游 `expires_in` 设置短生命周期。
- 刷新 Cookie 设置有限生命周期，刷新时轮换。
- 修改账号状态的请求拒绝跨站 `Origin` 或 `Sec-Fetch-Site: cross-site`。
- 不在日志中记录邮箱、验证码、访问令牌或刷新令牌。
- 上游错误转换为稳定的应用错误，不把供应商响应原文直接暴露给客户端。

## 配置

服务端仅在同时存在以下变量时启用账号：

- `SUPABASE_URL`
- `SUPABASE_ANON_KEY`

`SUPABASE_URL` 必须是 `https://`；仅在测试环境允许回环地址 `http://127.0.0.1`。密钥不写入仓库，也不由静态 GitHub Pages 暴露。

## 数据边界

- 邮箱、用户 ID 和认证令牌不写入 `state`、JSON 备份、CSV 或 Markdown 报告。
- 日常记录、训练、动作、模板、偏好和建议历史继续只在当前浏览器保存。
- 登录、退出、刷新会话都不读取或上传本机业务数据。
- 删除本地数据不等于删除 Supabase 账户；本阶段不提供账户删除，界面必须说明范围。
- 正式开放账号前，需要补充可执行的账户删除与支持渠道；当前未配置公开版不宣称账户已开放。

## 测试

- 未配置部署返回真实的本机模式。
- 配置缺一项时服务启动失败，避免半配置。
- 邮箱、验证码、请求方法和跨站来源校验。
- 模拟 Supabase 覆盖发码、验证、读取用户、刷新和退出。
- Cookie 包含 HttpOnly、SameSite、Path、合理 Max-Age，生产环境包含 Secure。
- 客户端覆盖未配置、未登录、等待验证码、已登录、错误和退出状态。
- 登录流程前后 `localStorage` 业务状态完全不变。
- 桌面与 390px 手机账号区域无横向溢出。
- 现有 PWA、离线、本地记录、JSON/CSV 迁移和云端建议测试继续通过。

## 发布边界

GitHub Pages 静态公开版将显示“本机模式”，因为它没有同源账号服务。Node/Render 部署在配置 Supabase 后才显示真实邮箱验证码入口。没有可工作的身份服务时绝不展示可提交的登录表单。
